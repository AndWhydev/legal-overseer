# Procedural Memory

Procedural Memory (`src/lib/knowledge-graph/procedural-memory.ts`) stores learned workflows as trigger-pattern-to-steps mappings. When a user message matches a stored trigger pattern, the procedure's steps are injected into the system prompt as a "Known Procedure" section.

## Schema

```typescript
interface ProceduralMemory {
  id: string
  org_id: string
  name: string              // e.g., "AWU email protocol"
  trigger_pattern: string   // Regex pattern to match against messages
  steps: string[]           // Ordered action sequence
  success_count: number     // Times executed successfully
  last_used_at: string | null
  source: 'observed' | 'explicit' | 'consolidation'
  is_active: boolean
}
```

## Creation Paths

Procedural memories can be created through three paths:

| Source | How It Works | Example |
|--------|-------------|---------|
| **Observed** | Sleep consolidation detects repeated tool sequences (3+ occurrences) and extracts the pattern | Agent always searches contacts then drafts email then sends via Outlook |
| **Explicit** | User instructs "remember how to do X" and the agent creates via `create_procedure` tool | "When I say send AWU email, always use Outlook not Gmail" |
| **Consolidation** | Promoted from `memory_patterns` table when confidence > 0.75 | Pattern of Monday morning inbox summaries becomes a procedure |

Trigger patterns are stored as regex strings and matched case-insensitively against the current message. The `matchProcedure()` function iterates all active procedures for the organization and returns the first match.

Invalid regex patterns (corrupted data) are silently skipped. A bad trigger pattern never crashes the system.
</Note>

```typescript
async function matchProcedure(
  supabase: SupabaseClient,
  orgId: string,
  message: string
): Promise<ProceduralMemory | null>
```

## Usage in Context Assembly

During context assembly, the assembler calls `matchProcedure()` with the current user message. If a procedure matches, its steps are formatted and injected into the system prompt:

```
## Known Procedure: AWU email protocol
1. Search contacts for the recipient
2. Draft email with professional tone
3. Send via Outlook (tor@allwebbedup.com.au)
4. Log activity with email details
```

This gives the agent a pre-defined playbook to follow, reducing errors on repetitive workflows.

## Success Tracking

Each time a procedure is triggered and the resulting tool sequence completes without errors, `incrementSuccess()` bumps the `success_count` and updates `last_used_at`. Procedures with high success counts are more likely to be surfaced in proactive recall.

<Tip>
Procedural memory is how BitBit gets better at repetitive tasks over time. The first time you do something, it learns. By the third time, it has a documented procedure. After that, it follows the procedure automatically.
</Tip>
