# TAOR Loop

The TAOR (Think, Act, Observe, Repeat) loop is BitBit's core execution engine. It replaced the original 1,100-line `engine.ts` with a clean, model-driven architecture where **the model decides when to stop** — no iteration caps, no convergence hints, no observation masking.

A safety ceiling of 50 iterations exists purely for runaway cost protection.

## Execution Flow

<Steps>
  <Step title="Pre-flight Checks">
    Before the loop begins, the engine runs validation:
    - Input length check (max 10,000 characters)
    - Thread ID validation
    - Injection detection and neutralization
    - Session authentication (Supabase Auth or Bearer token)
    - Calibrated confidence threshold loading (if available)
  </Step>
  <Step title="Model Routing">
    If no model is explicitly specified, the auto-router selects the appropriate tier:

    | Tier | Model | Max Tokens | Triggers |
    |------|-------|-----------|----------|
    | **Opus** | claude-opus-4 | 8192 | plan, strategy, analyze, compare, evaluate |
    | **Sonnet** | claude-sonnet-4 | 4096 | Standard CRUD, search, analysis |
    | **Haiku** | claude-haiku-4.5 | 2048 | classify, categorize, yes/no, tag, filter |

    The router checks for keyword matches (2+ required for promotion), message length heuristics, and multi-part question detection.
  </Step>
  <Step title="Context Assembly">
    The `ContextAssembler` builds the full context within a 48K token budget. This happens in parallel across four tiers with a 200ms latency target. See [Context Assembly](/intelligence/context-assembly) for details.
  </Step>
  <Step title="Tool Planning">
    The engine prepares the tool set:
    - **Eager tools** are always loaded (core CRUD, memory, search)
    - **Deferred tools** are resolved via Tool RAG — matching user intent against tool descriptions using vector similarity
    - Tool definitions follow Anthropic's tool-use format
  </Step>
  <Step title="Execution Loop">
    The core loop calls the Claude API and processes the response:

    ```
    Claude API Call → stop_reason?
      ├── tool_use → Execute tools → Feed results back → Loop
      └── end_turn → Yield final message → Done
    ```

    Tools execute in batches with streaming. Results are fed back as `tool_result` messages for the next iteration.
  </Step>
  <Step title="Response Guard">
    Before delivery, the response passes through:
    - **Leak detection** — scans for exposed API keys or credentials
    - **Leak scrubbing** — redacts any detected secrets
    - **Humanization** — ensures natural, non-robotic output
  </Step>
</Steps>

## SSE Event Stream

The loop yields events as a server-sent event stream:

| Event Type | Description |
|-----------|-------------|
| `stage` | Pipeline stage transitions (model_routing, context_assembly, etc.) |
| `thinking` | Model routing and planning information |
| `tool_call` | Tool invocation with name and parameters |
| `tool_result` | Tool execution result |
| `message` | Final text response (streamed incrementally) |
| `error` | Error information |
| `done` | Completion event with total token usage |

<Tip>
The `/api/agent/chat` route wraps the async generator in an SSE `ReadableStream`, sending each event as a `data:` frame. Clients can process events incrementally as they arrive.
</Tip>

## Circuit Breaker

External service calls (embedding APIs, channel adapters) are wrapped in a circuit breaker pattern. After repeated failures, the circuit opens and calls fail fast rather than cascading timeouts through the system. Failed operations are written to a dead letter queue for retry.
