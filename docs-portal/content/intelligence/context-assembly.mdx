# Context Assembly

The Context Assembler (`src/lib/context-assembly/context-assembler.ts`) builds all the context BitBit needs for each conversation turn. It targets **<200ms total assembly latency** by fetching all tiers in parallel with graceful per-tier degradation.

## Token Budget

The total context budget is **48,000 tokens**, managed by the `TokenBudgetManager`. Each tier receives a portion of the budget based on priority:

| Tier | Content | Budget Share | Fallback |
|------|---------|-------------|----------|
| **Working Memory** | System prompt, identity, guidelines, current state | Fixed (~4K) | Always loaded |
| **Session History** | Up to 20 recent turns verbatim + 40 compressed turns + key facts | ~20K | Truncates older turns |
| **Compiled Memory** | RAG results, Memory Palace recall, Knowledge Graph entities, procedural memories | ~18K | Returns empty |
| **Action State** | Pending approval queue items | ~2K | Skipped |

## Assembly Pipeline

The assembler runs these operations in parallel:

1. **Entity mention scanning** — detects contact/project names in the user message
2. **Session history loading** — recent messages + thread summaries
3. **Proactive recall** — graph-aware memory retrieval for mentioned entities
4. **Predictive context** — anticipatory loading based on deadlines, recency, and patterns
5. **Procedural memory matching** — checks trigger patterns against the current message
6. **Pending approvals** — loads any actions awaiting user confirmation

<Note>
Each fetch is wrapped in a `timedFetch` helper that catches errors and records latency. If any tier fails, the others still contribute their context — the system never fails completely due to a single tier timeout.
</Note>

## System Prompt Construction

The system prompt is built by `buildEntityAwarePrompt()` which composes:

- Base identity and behavioral guidelines
- Organization-specific settings and preferences
- Entity baseplates — structured snapshots for each mentioned contact
- Proactive recall results formatted as `<memory-context>` blocks
- Matched procedural memories as "Known Procedure" sections
- Pending action summaries

## Token Budget Manager

The `TokenBudgetManager` tracks allocation across tiers:

```typescript
interface TokenAllocation {
  tier: string
  allocated: number
  used: number
  remaining: number
}
```

Each tier reports its actual token usage after assembly. The manager ensures no tier exceeds its allocation, truncating content when necessary.

## Configuration

Default settings (overridable per-request):

```typescript
{
  tokenBudget: 48000,
  maxRecentTurns: 20,
  maxCompressedTurns: 40,
  maxEntities: 5,
  systemPromptCacheTtlMs: 300_000,  // 5 minutes
  includePendingActions: true,
  includeCompressedHistory: true,
}
```

<Tip>
The `systemPromptCacheTtlMs` setting caches the base system prompt for 5 minutes, avoiding redundant prompt construction on rapid-fire messages within the same conversation.
</Tip>
