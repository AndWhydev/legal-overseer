# ADR-0002: Dual Embedding Strategy

**Status:** Accepted

## Context

The entity knowledge graph needs vector embeddings for similarity search. BitBit already uses Voyage-3.5 (1024d) for channel message embeddings in Pinecone. The Cognitive Memory OS design calls for format-agnostic retrieval that can eventually handle text, PDFs, images, and other modalities.

## Decision

<Note>
Dual vector columns on `entity_nodes`:
- **Primary**: Google text-embedding-004 (768d) for multimodal-ready embedding
- **Secondary**: Voyage-3.5 (1024d) for high-precision text retrieval
</Note>

## Rationale

Google's embedding model supports multimodal inputs (text, images, PDFs) in a shared vector space. This enables future queries like "find the invoice Steve sent" to retrieve both the PDF attachment and the email discussing it.

Voyage-3.5 remains the stronger pure-text model, so it's kept as a secondary for text-heavy search paths.

pgvector supports multiple vector columns natively, so this costs nothing architecturally — just two HNSW indexes and two embedding API calls per entity.

## Consequences

**Positive:**
- Future-proof for multimodal ingestion (images, PDFs, etc.)
- No precision sacrifice on text workload (Voyage fallback)
- Embedding model choice is a config option, not a schema change

**Negative:**
- `GOOGLE_API_KEY` not yet configured (768d column starts as NULL, Voyage works as fallback)
- Two embedding API calls per entity node (cost: negligible at current scale)
- Two HNSW indexes consume more storage

## Configuration

Embedding models are configurable via environment variables:

```
EMBEDDING_MODEL_PRIMARY=text-embedding-004   # Google 768d
EMBEDDING_MODEL_SECONDARY=voyage-3.5         # Voyage 1024d
```

The system gracefully falls back to Voyage-only when the Google API key is not configured.
