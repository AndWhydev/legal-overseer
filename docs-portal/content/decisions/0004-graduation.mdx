# ADR-0004: Autonomy Graduation Mechanism

**Status:** Accepted

## Context

BitBit's tools have static autonomy levels. High-stakes tools like `send_email` are fixed at L2 (Propose First), meaning every email requires explicit user approval. This is safe but creates friction as the user builds trust with the system.

The confidence calibrator already runs nightly and adjusts routing thresholds based on outcome data. The question: how should individual tools graduate to higher autonomy levels?

## Decision

<Note>
Implement automatic **autonomy graduation** where tools promote from L2 (Propose) to L3 (Notify) when they accumulate **20+ approved outcomes** with **95%+ approval rate** over the last 30 days.
</Note>

## Mechanism

1. Nightly calibration cron checks per-tool approval rates in `action_outcomes`
2. Tools matching graduation criteria get promoted via `organisations.settings.autonomy_overrides`
3. `getAutonomyLevel()` checks org overrides first, so promotion takes effect immediately
4. **Safety cap**: Tools can only graduate to L3 (Act + Notify), never L4 (Silent)

## Graduation Candidates

Only high-impact L2 tools are eligible:
- `send_email`
- `send_gmail`
- `send_outlook`
- `send_sms`

## Rationale

- **Data-driven trust**: The system proves it makes good decisions before gaining autonomy
- **Per-org**: Different users may have different trust levels
- **Reversible**: An admin can remove the override to demote back to L2
- **L3 ceiling**: Email sending always notifies the user, even when autonomous
- **20 samples / 95% rate**: Borrowed from the calibrator's own thresholds

## Safeguards

- Seeded data is excluded (`threshold_source='seed'`). Only organic outcomes count
- A run of approved low-quality emails could graduate the tool, but the `was_correct` flag provides a secondary quality signal
- L3 always notifies the user, so graduated tools are never invisible
- Users can manually revoke graduation via org settings

## Consequences

**Positive:**
- BitBit becomes more autonomous over time without code changes
- User friction decreases as trust is established
- Graduation is transparent (logged, visible in org settings)

**Negative:**
- Premature graduation is possible if approval data is skewed (mitigated by organic-only filter)
- Users may not realize the system graduated (mitigated by L3 always notifying)
