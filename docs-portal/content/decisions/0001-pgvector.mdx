# ADR-0001: pgvector Entity Graph Over Pinecone

**Status:** Accepted

## Context

BitBit's memory retrieval was limited to flat vector search via Pinecone (Voyage-3.5, 1024d) with no relationship awareness. The system could not answer multi-hop questions like "How does Steve's project relate to Maya's delay?" because memories were stored as isolated documents.

Three storage options were evaluated:

1. **Keep Pinecone** + add a separate Postgres graph layer
2. **pgvector in Supabase** (co-located with existing data)
3. **Neo4j** or dedicated graph DB (purpose-built)

## Decision

<Note>
Use **pgvector in Supabase** for the entity knowledge graph. Keep Pinecone for channel_messages search only (existing pipeline, not migrated).
</Note>

## Decision Drivers

- Co-located graph+vector queries in a single SQL statement
- No additional external service or billing
- Bi-temporal edge model requires relational storage
- BitBit's scale is thousands of entities, not millions
- Supabase includes pgvector at no extra cost

## The Killer Feature: Co-located Queries

With pgvector, graph traversal and vector similarity happen in one query:

```sql
SELECT en.name, en.properties, ee.relation_type,
       en.embedding <=> $query_embedding AS distance
FROM entity_nodes en
JOIN entity_edges ee ON ee.target_id = en.id
WHERE ee.source_id = $steve_id
  AND ee.valid_until IS NULL
  AND en.embedding <=> $query_embedding < 0.5
ORDER BY distance LIMIT 10;
```

With Pinecone, this requires two round-trips (vector search + graph lookup) plus client-side join, adding 100ms+ latency.

## Consequences

**Positive:**
- Graph traversal + vector similarity in ~20ms (vs ~150ms with Pinecone)
- Single database for graph + vectors + application data
- HNSW indexes provide good recall at BitBit's scale
- Transactional consistency between edges and embeddings

**Negative:**
- pgvector has lower throughput than Pinecone at billion-scale (irrelevant for BitBit)
- Two vector systems in parallel during transition
- Must manage HNSW index parameters manually

## Implementation

- Dual vector columns: `embedding vector(768)` + `text_embedding vector(1024)`
- `match_entity_nodes` RPC function for cosine similarity
- HNSW indexes with `vector_cosine_ops` (m=16, ef_construction=64)
