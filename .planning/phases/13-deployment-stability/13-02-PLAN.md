     - `connectionTimeout`: 5000 (5 seconds max wait for connection)
     - `idleTimeout`: 30000 (30 seconds idle before release)
   - Export `POOL_RECOMMENDATIONS` object documenting:

     - Pro tier: 60 direct connections, more headroom

2. Create `personal-assistant/src/lib/supabase/service-client.ts`:
   - Export `getServiceClient()` that returns a singleton `SupabaseClient` using service role key
   - Use `createClient` from `@supabase/supabase-js` (NOT the SSR cookie-based client)
   - Configure with `SUPABASE_URL` (or `NEXT_PUBLIC_SUPABASE_URL`) and `SUPABASE_SERVICE_ROLE_KEY`
   - Set `auth: { autoRefreshToken: false, persistSession: false }` (service role doesn't need auth)
   - Use `db: { schema: 'public' }`
   - Add `global: { headers: { 'x-connection-pool': 'bitbit-service' } }` for pool identification
   - Singleton pattern: create once, reuse across requests in the same serverless instance
   - Add JSDoc explaining: this client is for cron jobs, agent processing, and background tasks where there is no user session. Dashboard/API routes serving user requests should continue using the cookie-based server.ts client.

Note: Supabase's JS client uses REST API by default (not direct Postgres connections), so connection pooling primarily matters for the Supabase infrastructure side. The service client singleton prevents creating multiple REST clients per invocation, reducing cold start overhead. Document that if direct Postgres access is needed later (e.g., for transactions), the pooler URL (port 6543) should be used.
