    Build the server-side push notification infrastructure.

    1. Create DB migration `20260328100000_push_tokens.sql`:
       ```sql
       CREATE TABLE push_tokens (
         id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
         user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
         token TEXT NOT NULL,
         platform TEXT NOT NULL CHECK (platform IN ('ios', 'android')),
         created_at TIMESTAMPTZ DEFAULT NOW(),
         updated_at TIMESTAMPTZ DEFAULT NOW(),
         UNIQUE(user_id, token)
       );

       CREATE INDEX idx_push_tokens_user ON push_tokens(user_id);

       ALTER TABLE push_tokens ENABLE ROW LEVEL SECURITY;

       CREATE POLICY "Users manage own tokens"
         ON push_tokens FOR ALL
         USING (auth.uid() = user_id)
         WITH CHECK (auth.uid() = user_id);

       -- Service role can read all tokens for dispatching
       CREATE POLICY "Service role reads all tokens"
         ON push_tokens FOR SELECT
         USING (auth.role() = 'service_role');
       ```

    2. Create `personal-assistant/src/lib/notifications/push-dispatcher.ts`:
       - sendPushNotification(tokens: string[], notification: { title, body, data?, badge? }):
         POST to https://exp.host/--/api/v2/push/send
         Body: tokens.map(to => ({ to, title, body, data, badge, sound: 'default' }))
         Handle response: check each ticket for errors, collect DeviceNotRegistered tokens for cleanup
       - sendPushToUser(userId: string, notification): looks up tokens from push_tokens, calls sendPushNotification
       - cleanupInvalidTokens(tokens: string[]): DELETE FROM push_tokens WHERE token IN (...)
       - Uses service-role Supabase client for DB access
       - Wraps all external calls in try/catch -- push failure should never break the calling flow

    3. Write tests in push-dispatcher.test.ts:
       - Mock fetch for Expo Push API
       - Mock Supabase for token lookup
       - Verify payload format matches Expo Push API spec
       - Verify graceful handling of no tokens, invalid tokens
       - Verify batch sending (single API call for multiple tokens)

    4. Create `personal-assistant/src/app/api/push/register/route.ts`:
       - POST handler: accepts { token: string, platform: 'ios' | 'android' }
       - Authenticates via Bearer token (import authenticateBearer from Plan 01)
       - Falls back to cookie auth via createClient
       - Upserts to push_tokens table (conflict on user_id + token)
       - Returns 200 on success

    5. Wire push into approval queue:
       - In createApproval (approval-queue.ts), after DB insert, call sendPushToUser
       - Notification: title="Approval Needed", body=approval.title, data={ type: 'approval', id: approval.id }
       - Import is lazy (dynamic import) to avoid circular deps and keep push optional
       - Wrap in try/catch -- approval creation must not fail if push fails

    6. Wire push into workflow engine:
       - In executeWorkflow, on workflow completion, call sendPushToUser
       - Notification: title="Workflow Complete", body=`${rule.name} finished`, data={ type: 'workflow', id: run.id }
       - Same lazy import + try/catch pattern
  </action>
