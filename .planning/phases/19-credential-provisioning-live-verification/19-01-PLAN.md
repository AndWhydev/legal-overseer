  <action>
    Create a standalone Fly.io deployment for the WhatsApp Baileys bridge at `deployments/whatsapp-bridge/`. This is separate from the existing `deployments/fly/` agent worker because the bridge needs persistent WebSocket connections.

    **fly.toml:**
    - App name: `bitbit-whatsapp-bridge`
    - Region: `syd` (same as other deployments)
    - `min_machines_running = 1` (must always be on for WhatsApp connection)
    - `auto_stop_machines = "off"` (never stop -- bridge must stay connected)
    - Health check at `/health` every 30s
    - Mount volume `bridge_data` at `/data` for auth state persistence
    - shared-cpu-1x with 512mb memory (Baileys is lightweight)

    **package.json:**
    - Dependencies: `@whiskeysockets/baileys`, `@supabase/supabase-js`, `pino` (Baileys peer dep)
    - Scripts: `build`, `start`, `dev`
    - TypeScript compilation target

    **Dockerfile:**
    - Node 20 slim base
    - Copy and install deps, compile TS, run dist/server.js
    - Volume mount for `/data`

    **src/bridge-manager.ts:**
    - Adapts the existing `BaileysBridge` class pattern from `personal-assistant/src/lib/channels/baileys-bridge.ts`
    - Creates Supabase client using service role key (no user auth in standalone worker)
    - Manages bridge lifecycle: start on boot, reconnect on disconnect, persist auth state to both filesystem (`/data/auth_state/`) and Supabase `whatsapp_sessions` table
    - Exports `startBridge(orgId)`, `stopBridge(orgId)`, `getBridgeStatus(orgId)`
    - On message receive: insert into `channel_messages` table via Supabase REST API
    - Outbox drain loop: poll `whatsapp_outbox` every 5s, send pending messages
    - Health reporting: log to `channel_health` table every 60s
    - Max reconnect: 5 attempts with exponential backoff (5s, 15s, 45s, 135s, 405s), then alert via Supabase insert to `notifications` table

    **src/server.ts:**
    - Plain Node.js HTTP server (same pattern as `deployments/fly/src/worker.ts`)
    - Routes:
      - `GET /health` -- returns bridge status JSON (connected/disconnected, session age, last activity)
      - `POST /bridge/start` -- starts bridge for org (requires BRIDGE_SECRET header auth)
      - `POST /bridge/stop` -- stops bridge for org
      - `GET /bridge/status` -- returns detailed status
      - `GET /bridge/qr` -- returns current QR code if in pairing state
    - Auto-start bridge on server boot using `DEFAULT_ORG_ID` env var
    - Bearer token auth via `BRIDGE_SECRET` env var on all POST routes

    **Environment variables needed (document in comments):**
    - SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY (database access)
    - DEFAULT_ORG_ID (auto-start bridge for this org)
    - BRIDGE_SECRET (auth for management endpoints)

  <action>
    Create a comprehensive step-by-step runbook at `docs/credential-provisioning-runbook.md` documenting how to provision credentials for all OAuth providers. This is the key deliverable for OAUTH-01, OAUTH-02, OAUTH-04, OAUTH-05.

    Structure the runbook with these sections:

    **1. Google Cloud (Gmail OAuth) -- OAUTH-01, CHAN-01**
    - Go to Google Cloud Console > APIs and Services > Credentials
    - Create OAuth 2.0 Client ID (Web application type)
    - Add authorized redirect URI: `https://{APP_DOMAIN}/callback/gmail`
    - Enable Gmail API in API Library
    - Copy Client ID and Client Secret
    - Set in Vercel: `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`
    - Verification: Navigate to Settings > Channels > Gmail > Connect, complete OAuth flow
    - Troubleshooting: common errors (redirect_uri_mismatch, access_denied, etc.)

    **2. Microsoft Azure AD (Outlook OAuth) -- OAUTH-02, CHAN-02**
    - Go to Azure Portal > App registrations > New registration
    - Set redirect URI: `https://{APP_DOMAIN}/callback/outlook` (Web platform)
    - Under API permissions, add Microsoft Graph: Mail.Read, Mail.Send, offline_access
    - Grant admin consent
    - Under Certificates & secrets, create new client secret
    - Copy Application (client) ID, Directory (tenant) ID, client secret value
    - Set in Vercel: `OUTLOOK_CLIENT_ID`, `OUTLOOK_CLIENT_SECRET`, `OUTLOOK_TENANT_ID`
    - Verification: Navigate to Settings > Channels > Outlook > Connect, complete OAuth flow
    - Note: For multi-tenant, set "Supported account types" to "Accounts in any organizational directory"

    **3. Asana OAuth -- OAUTH-04**
    - Go to Asana Developer Console (https://app.asana.com/0/developer-console)
    - Create new app
    - Set redirect URI: `https://{APP_DOMAIN}/callback/asana`
    - Copy Client ID and Client Secret
    - Set in Vercel: `ASANA_CLIENT_ID`, `ASANA_CLIENT_SECRET`
    - Verification: Navigate to Settings > Channels > Asana > Connect, complete OAuth flow

    **4. Calendly OAuth -- OAUTH-05**
    - Go to Calendly Developer Portal (https://developer.calendly.com)
    - Create OAuth application
    - Set redirect URI: `https://{APP_DOMAIN}/callback/calendly`
    - Copy Client ID and Client Secret
    - Set in Vercel: `CALENDLY_CLIENT_ID`, `CALENDLY_CLIENT_SECRET`
    - Verification: Navigate to Settings > Channels > Calendly > Connect, complete OAuth flow

    **5. WhatsApp Bridge Deployment -- CHAN-03**
    - Prerequisites: Fly.io CLI installed, authenticated
    - `cd deployments/whatsapp-bridge`
    - `fly apps create bitbit-whatsapp-bridge`
    - `fly volumes create bridge_data --region syd --size 1`
    - `fly secrets set SUPABASE_URL=... SUPABASE_SERVICE_ROLE_KEY=... DEFAULT_ORG_ID=... BRIDGE_SECRET=...`
    - `fly deploy`
    - Verify: `fly status` shows machine running
    - Pair: `curl https://bitbit-whatsapp-bridge.fly.dev/bridge/qr` to get QR, scan with phone
    - Monitor: `fly logs` to check connection stability
