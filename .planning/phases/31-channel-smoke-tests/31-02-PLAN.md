    - Test: processBatch continues processing after individual item failure
    - Test: processBatch collects all errors and returns them alongside successes
    - Test: processBatch dead-letters items that fail after retries (when supabase client provided)

  <action>
**Create cron-resilience.ts** with these utilities:

1. `withRetry<T>(fn: () => Promise<T>, opts: { maxRetries: number; baseDelayMs: number; label: string }): Promise<T>`
   - Retries with exponential backoff: delay = baseDelayMs * 2^attempt
   - Catches errors, logs retry attempt with label
   - After maxRetries, throws the last error
   - Detects "connection" and "timeout" errors for retry eligibility (not all errors should retry)

2. `withBackoff<T>(fn: () => Promise<T>, opts: { label: string }): Promise<T>`
   - Wraps fn, catches rate limit errors (status 429, message containing "rate limit" or "too many requests")
   - On rate limit: waits Retry-After header value (or 60s default), then retries once
   - Non-rate-limit errors pass through

3. `processBatch<T, R>(items: T[], processor: (item: T) => Promise<R>, opts: { label: string; circuitKey?: string; supabase?: SupabaseClient; orgId?: string }): Promise<{ results: R[]; errors: { item: T; error: string }[] }>`
   - Processes items sequentially (not parallel -- respects rate limits)
   - Each item wrapped in try/catch
   - If circuitKey provided, checks `getCircuitState(circuitKey)` before each item -- skips remaining if open
   - Failed items: if supabase + orgId provided, dead-letter them. Always add to errors array.
   - Returns both successful results and error details.

**Update cron-guard.ts:**
- Add optional `resilience` option to `withCronGuard` signature: `withCronGuard(request, handler, opts?: { resilience?: boolean })`
- When `resilience: true` (or by default for all cron routes), wrap the handler call in a top-level try/catch that:
  - On DB connection errors (message contains "connection", "ECONNREFUSED", "timeout"): retry once after 2s delay
  - On any error: log to Sentry (already happens), return 500 with error detail (already happens)
- This is a minimal change -- the real resilience is in `processBatch` which individual cron routes opt into.
