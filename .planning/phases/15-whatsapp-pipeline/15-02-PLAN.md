**Approval flow hardening (WHATS-03):**

1. In `conversation-manager.ts`, update `handleApprovalDecision` and `handleApproveIntent`:
   - Add retry logic: if `resolveApproval` throws a transient error (network/timeout), retry once after 1 second before reporting failure
   - Add idempotency guard: if approval is already resolved (APPROVAL_ALREADY_RESOLVED), respond with "Already handled" instead of generic error
   - After successful approval resolution, log an audit entry:
     ```ts
     console.log(JSON.stringify({
       event: 'whatsapp_approval',
       orgId: state.orgId,
       approvalId: approval.id,
       decision,
       source: 'whatsapp',
       latencyMs: Date.now() - startMs,
     }))
     ```

2. In `handleConfirmation`, ensure Y/N matching is robust:
   - Already handles variations (yes/yep/yeah/ok/confirm/go/sure and no/nah/nope/cancel/stop/nevermind)
   - Add unicode variations that WhatsApp users sometimes send: thumbs up emoji (👍) as yes, thumbs down (👎) as no
   - Add "approved"/"rejected" as explicit keywords (since approval notifications use these words)

3. Edge case: if a user sends "Y" but there are no pending approvals AND no pending confirmation:
   - Currently goes to `handleNewMessage` which parses as approval intent then checks queue — this is correct
   - Add a fast-path: in `handleApproveIntent`, if no pending approvals, check if there are any expired approvals in last hour and inform user: "That approval has expired. Check dashboard for details."

**Latency instrumentation (WHATS-04):**

4. In `conversation-manager.ts`, wrap `handleIncomingMessage` with timing:
   - Record `startMs = Date.now()` at entry
   - After reply is sent, log structured JSON:
     ```ts
     console.log(JSON.stringify({
       event: 'whatsapp_e2e_latency',
       orgId,
       intentDetected: command?.intent || 'none',
       totalMs: Date.now() - startMs,
       source: 'whatsapp',
       isVoiceNote: text.startsWith('[Voice note]'),
     }))
     ```
   - This covers the full pipeline: receive -> parse -> dispatch -> reply

5. In `whatsapp/route.ts` (Meta Cloud API webhook), add timing for the webhook processing:
   - Record wall-clock time at start of POST handler
   - After `processWhatsAppMessage` completes (change from fire-and-forget to tracked):
     - Keep the fire-and-forget `.catch()` pattern but add `.finally()` to log timing
     - Add `X-WhatsApp-Process-Ms` header to the response (informational)
   - Log structured JSON with `event: 'whatsapp_webhook_latency'` including message type (text/audio)

6. In `baileys-bridge.ts` (from Plan 01), add timing to the message receive handler:
   - Record timestamp when Baileys `messages.upsert` fires
   - After `processWhatsAppMessage` completes, log `event: 'whatsapp_bridge_latency'` with phase breakdown:
     - `receiveMs`: time from Baileys event to message parsed
     - `transcribeMs`: time for Whisper (0 for text, >0 for voice)
     - `insertMs`: time for channel_messages insert
     - `processMs`: time for processWhatsAppMessage
     - `totalMs`: end-to-end

All latency logs use structured JSON matching the relay-daemon pattern (event, channel, durations) for consistent log aggregation.
