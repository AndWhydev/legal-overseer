Create `baileys-bridge.ts` — a Baileys connection manager class `BaileysBridge`:

1. **Connection lifecycle:**
   - `createBridge(supabase, orgId)` factory that creates a Baileys WASocket connection
   - QR code generation: on `connection.update` event with QR, store QR string in `whatsapp_sessions` table (status='pairing', qr_code column)
   - On successful connection: update session status to 'connected', store auth creds to `whatsapp_sessions.auth_state` (JSON blob)
   - On disconnect: update session status to 'disconnected', log to channel_health via existing `logSessionHealth`
   - Auth state persistence: on `creds.update` event, save creds to Supabase `whatsapp_sessions.auth_state` for session resume

2. **Message receive loop:**
   - On `messages.upsert` event, for each incoming message:
     - Extract sender (jid), text content, and message type (text, audio, image, etc.)
     - For text messages: insert into `channel_messages` with channel='whatsapp', external_id=message key, sender=push name or jid
     - For audio messages: download media using Baileys `downloadMediaMessage`, then call `transcribeVoiceNote` (from Task 2's whatsapp-voice.ts), insert transcribed text into channel_messages with metadata.voice_note=true
     - After insert: trigger `processWhatsAppMessage` from whatsapp-parser.ts (same flow as webhook)
   - Dedup: use `isDuplicate` from existing dedup.ts before inserting

3. **Message send (outbox drain):**
   - Poll `whatsapp_outbox` for pending messages every 5 seconds (matches existing `sendMessageViaBridge` pattern in whatsapp.ts)
   - Send via Baileys sock.sendMessage, update outbox row status to 'sent' or 'failed'

4. **Health and reconnection:**
   - Track connection age and last activity (reuse WhatsAppSessionStatus interface from whatsapp-monitor.ts)
   - On unexpected disconnect, attempt reconnect up to 3 times with exponential backoff (5s, 15s, 45s)
   - Log reconnection attempts to console with structured JSON

5. **Import Baileys as optional dependency:** Use dynamic import `await import('@whiskeysockets/baileys')` so the module doesn't fail at import time if Baileys isn't installed. Export a `isBaileysAvailable()` check.

Create `bridge/route.ts` — HTTP management endpoint:
- `POST`: Start bridge for org (reads org_id from auth session). Creates bridge instance, returns session_id. If session already exists and is connected, return current status.
- `GET`: Return bridge status (connected/disconnected/pairing) and QR code if in pairing state. Include session age and last activity from whatsapp_sessions.
- Both routes use `createClient` from `@/lib/supabase/server` and require auth.

**WHATS-05 documentation:** Add a JSDoc block at the top of baileys-bridge.ts documenting:
- Baileys pros: no Meta Business verification needed, works immediately, free, full message types
- Baileys cons: unofficial, can break with WA updates, requires persistent process, phone must stay online
- Cloud API pros: official, reliable, webhook-based (no persistent process), phone can be offline
- Cloud API migration path: swap baileys-bridge.ts for cloud-api-bridge.ts, same interfaces (message receive -> channel_messages, outbox drain -> send), webhook route already handles Cloud API payloads
- Recommendation: use Baileys for dev/MVP, migrate to Cloud API when Meta Business verification complete

Do NOT install @whiskeysockets/baileys via npm in this task — the dynamic import pattern means the module works without it (returns unavailable). Installation is a user setup step.

Create `whatsapp-voice.ts` — voice note transcription service:

1. `transcribeVoiceNote(audioBuffer: Buffer, mimeType: string): Promise<string | null>`:
   - Reuse the Whisper transcription logic from the existing `/api/ai/voice/route.ts` (same pattern: create FormData, POST to OpenAI Whisper API, return text)
   - Accept raw audio buffer and MIME type (WhatsApp sends audio/ogg;codecs=opus)
   - Return transcribed text or null on failure
   - Log errors with `[whatsapp-voice]` prefix
   - If OPENAI_API_KEY is not set, return null with warning log

2. `downloadWhatsAppMedia(mediaId: string, accessToken: string): Promise<Buffer | null>`:
   - For Meta Cloud API: GET media URL from `graph.facebook.com/v21.0/{mediaId}`, then download binary
   - Return buffer or null on failure
   - This is for the Cloud API webhook path. The Baileys bridge uses Baileys' own `downloadMediaMessage` instead.

Update `whatsapp/route.ts` webhook handler:
- Currently only processes `msg.type === 'text'` — extend to also handle `msg.type === 'audio'`:
  - Download audio using `downloadWhatsAppMedia(msg.audio.id, accessToken)`
  - Transcribe using `transcribeVoiceNote(audioBuffer, msg.audio.mime_type)`
  - If transcription succeeds: insert into channel_messages with body=transcription, metadata.voice_note=true, metadata.original_media_id=msg.audio.id
  - Then call `processWhatsAppMessage` with the transcribed text (same as text messages)
  - If transcription fails: insert with body='[Voice note - transcription unavailable]', mark is_actionable=false

Update `whatsapp-parser.ts`:
- Add voice note awareness: if messageRow has metadata.voice_note=true, prepend "[Voice note] " to the text before passing to `handleIncomingMessage`. This helps the command parser understand the input is from speech (more casual, may need more lenient parsing).

Latency consideration: voice notes add ~2-3s for Whisper transcription. This is acceptable within the 10s SLA (WHATS-04) since the rest of the pipeline is <2s.
