  <implementation>
Create `personal-assistant/src/lib/brain/anomaly-detector.ts` with the following exports:

1. `MIN_SAMPLE_SIZE = 5` -- constant, gate for z-score computation
2. `ALERT_BUDGET_MAX = 3` -- constant, per entity per 24h
3. `Z_SCORE_ALERT_THRESHOLD = 3` -- constant, minimum z-score for alert
4. `Z_SCORE_CROSS_ENTITY_THRESHOLD = 2` -- constant, minimum for cross-entity inclusion
5. `CROSS_ENTITY_MIN_COUNT = 3` -- constant, minimum entities for pattern break

6. `computeZScore(value: number, baseline: AnomalyBaseline): number | null`
   - Returns null if baseline.sample_count < MIN_SAMPLE_SIZE
   - Returns null if baseline.stddev === 0
   - Returns zScore(value, baseline.mean, baseline.stddev) from simple-statistics

7. `updateBaseline(baseline: Pick<AnomalyBaseline, 'mean' | 'stddev' | 'sample_count'>, newValue: number): { mean: number; stddev: number; sample_count: number }`
   - Uses addToMean from simple-statistics for mean
   - Implements Welford's online variance update for stddev
   - Handles sample_count === 0 edge case (first value: mean = newValue, stddev = 0, count = 1)

8. `extractMetrics(entries: KnowledgeLogEntry[]): MetricExtraction[]`
   - Rule-based extraction (NO LLM calls):
     - `invoice` signal_type: extract payment_amount from `$X,XXX.XX` pattern in content; extract payment_timing from `N days late/overdue/after` pattern
     - `message` signal_type: emit { metric_name: 'message_frequency', value: 1 } per entity_id
     - `response_latency` is NOT extracted here -- it requires correlating timestamps across consecutive message pairs per entity, which needs a stateful window. Add a code comment: `// TODO(phase-47): response_latency extraction requires cross-entry timestamp correlation`
     - Skip entries with empty entity_ids array
   - Returns array of MetricExtraction

9. `async isWithinAlertBudget(supabase: SupabaseClient, orgId: string, entityId: string, maxAlerts?: number): Promise<boolean>`
   - Queries brain_alerts for count in last 24h for this org+entity
   - Returns (count < maxAlerts) where maxAlerts defaults to ALERT_BUDGET_MAX
   - Returns false on error (fail closed)

10. `async generateAnomalyExplanation(metricName: MetricName, value: number, baseline: Pick<AnomalyBaseline, 'mean' | 'stddev'>, zScore: number): Promise<string>`
    - Calls gateway(models.fast) with prompt: "Generate a concise one-sentence anomaly explanation for a personal assistant. Metric: {metricName}. Current value: {value}. Baseline mean: {baseline.mean}. Z-score: {zScore}. Format: brief natural language comparison."
    - Returns generated text or fallback string on error: `${metricName} anomaly detected (z=${zScore.toFixed(1)}, baseline: ${baseline.mean.toFixed(1)})`

11. `async detectAndAlertAnomalies(supabase: SupabaseClient, orgId: string, entityId: string, entries: KnowledgeLogEntry[]): Promise<{ anomaliesDetected: number; alertsSent: number }>`
    - Extract metrics from entries via extractMetrics
    - For each metric extraction:
      - Load or initialize baseline from anomaly_baselines table (upsert pattern)
      - Compute z-score
      - Update baseline with new value (upsert back)
      - If z-score > Z_SCORE_ALERT_THRESHOLD and isWithinAlertBudget:
        a. Generate explanation via generateAnomalyExplanation
        b. Insert alert record into brain_alerts table (with severity derived from z-score: z>=5 -> 'critical', z>=4 -> 'high', z>=3 -> 'medium')
        c. **Deliver alert to user** via dispatchNotification:
           ```typescript
           await dispatchNotification(supabase, {
             orgId,
             type: 'alert_escalation',
             title: `Anomaly: ${metricName}`,
             body: explanation,
             urgency: severity === 'critical' ? 'critical' : severity === 'high' ? 'high' : 'normal',
             metadata: { entityId, metricName, zScore: z, alertType: 'anomaly' },
           })
           ```
        d. Increment alertsSent only after both insert AND dispatch succeed
    - Returns counts
    - Wraps in try/catch, returns { anomaliesDetected: 0, alertsSent: 0 } on error (non-critical)

    IMPORTANT: The `alertsSent` counter reflects alerts that were both persisted to brain_alerts AND dispatched to the user. If the brain_alerts insert succeeds but dispatch fails, the alert is still recorded (for dashboard visibility) but alertsSent is NOT incremented for that alert. Log the dispatch failure at warn level.

12. `async detectCrossEntityPatternBreaks(supabase: SupabaseClient, orgId: string): Promise<{ breaksDetected: number; alertsSent: number }>`
    - Query recent brain_alerts of type 'anomaly' from last consolidation window (30 minutes)
    - Group by metric_name
    - If any metric has 3+ distinct entity_ids with z>2: generate pattern_break alert
    - Check alert budget before sending (use entity_id of first entity in group)
    - Insert pattern_break alert into brain_alerts
    - Dispatch pattern_break notification via dispatchNotification with urgency 'high' and type 'alert_escalation'

Import pattern:

Error handling: All async functions use try/catch with logger.warn('[anomaly-detector]...') and return graceful fallbacks. Never throw from this module. Dispatch failures are logged but do not prevent the alert from being recorded in brain_alerts.
