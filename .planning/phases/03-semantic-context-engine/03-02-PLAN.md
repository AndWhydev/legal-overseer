<feature>
  <name>5-Step Fuzzy Entity Resolution</name>
  <files>personal-assistant/src/lib/context/entity-resolver.ts, personal-assistant/src/lib/context/__tests__/entity-resolver.test.ts</files>
  <behavior>
    The resolver cascades through 5 steps, stopping at the first step that returns results:

    Step 1 — Exact alias match (confidence: 1.0):
      Input: "Sezer" → Query: `aliases.cs.{sezer}` (case-insensitive array contains)
      Expected: Returns contact with alias "sezer" or "Sezer"

    Step 2 — Email match (confidence: 0.95):
      Input: "sezer@example.com" → Query: `emails.cs.{sezer@example.com}`
      Expected: Returns contact with that email

    Step 3 — Phone match (confidence: 0.90):
      Input: "0412345678" → Query: `phones.cs.{0412345678}`
      Expected: Returns contact with that phone

    Step 4 — Partial name match (confidence: 0.70):
      Input: "Sez" → Query: `name.ilike.%sez%` OR `slug.ilike.%sez%`
      Expected: Returns contacts whose name or slug contains "sez"

    Step 5 — Phone variant match (confidence: 0.60):
      Input: "0412 345 678" → Normalize to "0412345678", also try "+61412345678"
      Query: phones array contains any variant
      Expected: Returns contact matched by phone normalization

    Each result includes a `matchConfidence` and `matchStep` field.
    Results are ranked by confidence (highest first).
    If multiple steps match, only the highest-confidence step's results are returned.

    `resolveEntity(query, orgId)` — backward-compatible, returns Contact[]
    `resolveEntityRanked(query, orgId)` — new, returns { contact: Contact, matchConfidence: number, matchStep: string }[]
  </behavior>
  <implementation>
    Rewrite entity-resolver.ts:
    1. Keep the existing `resolveEntity` export signature for backward compatibility
    2. Add `resolveEntityRanked` that returns ranked results
    3. Implement phone normalization: strip spaces/dashes/parens, handle AU format (0→+61)
    4. Each step is a separate async function that returns matches or empty array
    5. Cascade: try step 1, if empty try step 2, etc.
    6. Use existing Supabase `createClient` from `@/lib/supabase/server`
    7. Update `search_contacts` handler in tools.ts to use `resolveEntityRanked` instead of raw Supabase query

    Tests use mock Supabase client. Test each step independently:
    - Test exact alias: mock returns contact when alias matches
    - Test email: mock returns contact when email matches
    - Test phone: mock returns contact when phone matches
    - Test partial name: mock returns contacts when name partially matches
    - Test phone variants: mock returns contact when normalized phone matches
    - Test cascade: mock step 1 returns empty, step 2 returns result → returns step 2 result
    - Test no match: all steps return empty → returns []
  </implementation>
</feature>
