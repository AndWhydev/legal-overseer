   - `gmail`: clientId from GOOGLE_CLIENT_ID, clientSecret from GOOGLE_CLIENT_SECRET, authorizationUrl `https://accounts.google.com/o/oauth2/v2/auth`, tokenUrl `https://oauth2.googleapis.com/token`, scopes `['https://mail.google.com/', 'https://www.googleapis.com/auth/gmail.readonly', 'https://www.googleapis.com/auth/gmail.send']`, supportsPKCE: true. Add `access_type: 'offline'` and `prompt: 'consent'` to the params in getOAuthRedirectUrl for gmail specifically (needed to get refresh_token from Google).
   - `outlook`: clientId from OUTLOOK_CLIENT_ID, clientSecret from OUTLOOK_CLIENT_SECRET, authorizationUrl `https://login.microsoftonline.com/common/oauth2/v2.0/authorize`, tokenUrl `https://login.microsoftonline.com/common/oauth2/v2.0/token`, scopes `['https://graph.microsoft.com/Mail.Read', 'https://graph.microsoft.com/Mail.Send', 'offline_access']`, supportsPKCE: true.

   Note: `asana` and `calendly` already exist in PROVIDERS.

2. In `getOAuthRedirectUrl`, after building the base params, add provider-specific params: if provider is `gmail`, add `access_type=offline` and `prompt=consent` to the URLSearchParams.

3. In `types.ts` (integrations), update AVAILABLE_INTEGRATIONS:

   - Change `whatsapp` status from `coming_soon` to `available`, authMethod stays `api_key` (will use QR pairing in practice, but the type is fine).

4. Create migration `052_channel_oauth_expansion.sql`:
   - ALTER TABLE channel_connections: drop and recreate the CHECK constraint on channel_type to include: 'gmail', 'outlook', 'whatsapp', 'asana', 'calendly', 'stripe', 'imessage', 'calendar', 'reminders', 'gsc'.
   - ALTER TABLE channel_messages: same expansion for the channel CHECK constraint.
   - Add a `content_hash` TEXT column to channel_messages (for cross-channel dedup).
   - Add index: `CREATE INDEX IF NOT EXISTS idx_channel_messages_content_hash ON channel_messages(content_hash) WHERE content_hash IS NOT NULL;`

5. Update `callback/[provider]/route.ts`:
   - The existing callback is generic and already handles any provider via `exchangeOAuthCode(provider, code, codeVerifier)`. This works for gmail/outlook without code changes to the exchange logic.
   - Change the success redirect from `/dashboard/settings?connected=PROVIDER` to `/dashboard/channels?connected=${encodeURIComponent(provider)}` so the OAuth completion lands on the channel settings page where Plan 03's UI listens for the `?connected=` query param.
   - Change error redirects similarly: from `/dashboard/settings?error=...` to `/dashboard/channels?error=...`.
   - Also upsert a `channel_connections` row with status='connected' after storing credentials, so the status API (Plan 01 Task 2) immediately reflects the new connection.
