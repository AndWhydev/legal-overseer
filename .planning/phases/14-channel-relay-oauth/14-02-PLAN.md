  <action>
1. Create `POST /api/channels/connect`:
   - Authenticate user via supabase.auth.getUser(), get org_id from profiles table.
   - Accept JSON body: `{ channel: string, credentials?: Record<string, string> }`.
   - For OAuth channels (gmail, outlook, asana, calendly): return `{ redirect: true, url: string }` where url is the OAuth start URL (`/api/auth/oauth/start?provider=${channel}`). The existing OAuth callback at `/callback/[provider]` handles token exchange and storage.
   - For API key channels (stripe): expect `credentials.secret_key` in body. Encrypt and store via `storeOrgCredential(supabase, orgId, channel, credentials, userId)`. Also upsert a `channel_connections` row with status='connected'.
   - For WhatsApp: return `{ pairing: true, sessionId: string }` -- insert a whatsapp_sessions row with status='pairing' and return its id. The QR code flow is handled by the bridge worker (Phase 15 concern), but we set up the DB row here.
   - Return 200 with the result.

2. Create `POST /api/channels/disconnect`:
   - Authenticate user, get org_id.
   - Accept JSON body: `{ channel: string }`.
   - Call `deleteOrgCredential(supabase, orgId, channel)` to remove credentials.
   - Update `channel_connections` set status='disconnected', relay_enabled=false for this org+channel.
   - Per user decision: keep historical messages (do NOT delete from channel_messages).
   - Return 200.

3. Create `GET /api/channels/[channel]/config`:
   - Authenticate user, get org_id.
   - Extract channel from route params.
   - Query `channel_connections` for this org+channel -- return config, poll_interval_seconds, relay_enabled.
   - Also accept PATCH to update config (sync frequency, relay_enabled toggle, filters).
   - Return the channel config object.

Note: These 3 routes are tightly coupled (connect/disconnect/config for a single channel entity) and share the same auth+org lookup pattern, so they are grouped in one task for coherence. Total is ~150 lines across 3 files.
  </action>

  <action>
1. Create `token-refresh.ts` with these exports:

   `refreshChannelToken(supabase: SupabaseClient, orgId: string, channel: string): Promise<{ refreshed: boolean; error?: string }>`:
   - Read credentials from org_integrations for this org+channel.
   - Decrypt credentials. Check if token_expires_at is within 15 minutes of expiry.
   - If not expiring, return { refreshed: false } (no action needed).
   - For gmail: use Google's token endpoint with refresh_token grant. Store new access_token and token_expires_at back to org_integrations via storeOrgCredential.
   - For outlook: use Microsoft's token endpoint with refresh_token grant. Store new tokens.
   - For asana: use Asana's token endpoint with refresh_token grant.
   - For calendly: use Calendly's token endpoint with refresh_token grant.
   - For stripe: API keys don't expire, return { refreshed: false }.
   - On refresh failure: increment a retry_count in the credentials metadata. After 24 retries (24-hour grace period at hourly checks), update org_integrations status to 'error'. Per user decision, this should trigger a dashboard warning badge AND an email alert.
   - When status transitions to 'error': send email notification to org owner using the existing notification dispatcher (`src/lib/notifications/dispatcher.ts`). Use template: "Your {channel} connection needs re-authorization. Click here to reconnect." Include a link to `/dashboard/channels`.
   - Return { refreshed: true } on success.

   `refreshAllTokens(supabase: SupabaseClient): Promise<{ results: Array<{ orgId: string; channel: string; refreshed: boolean; error?: string }> }>`:
   - Query org_integrations for all rows with status='connected'.
   - Call refreshChannelToken for each.
   - Return aggregated results.

2. Create the cron route `personal-assistant/src/app/api/cron/token-refresh/route.ts`:
   - Use the same cron guard pattern from Phase 13 (check CRON_SECRET header).
   - Call refreshAllTokens with service-role client.
   - Return results summary as JSON.
  </action>
