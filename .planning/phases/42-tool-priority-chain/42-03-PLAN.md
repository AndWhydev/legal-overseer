# Plan 42-03: Human Handoff â€” Synchronous Mid-Execution Gate

<objective>

Extends the existing approval queue pattern (CONTEXT.md D-07) and integrates with the async task lifecycle from Phase 39. This addresses CHAIN-04: human handoff as synchronous mid-execution gate.

2. **Handler function**:
```typescript
export async function handleHumanHandoff(
  input: {
    description: string
    context?: {
      service?: string
      attempted_tiers?: string[]
      reason?: string
      url?: string
    }
    urgency?: 'urgent' | 'normal' | 'low'
    expected_result: string
  },
  orgId: string,
  supabase: SupabaseClient,
  execOptions?: { agentConfigId?: string; agentRunId?: string }
): Promise<ToolResult>
```

The handler:
a. Creates an approval record via `queueAgentAction()` with:
   - `action_type: 'human_handoff'`
   - `action_summary`: the `description` field
   - `action_payload`: full input including context, expected_result
   - `confidence_score: 0` (always requires human, never auto-approve)
   - `priority`: maps from `urgency` (urgentâ†’urgent, normalâ†’normal, lowâ†’low)
   - `routing_decision: 'ask'`

b. Dispatches notification via `notifyApproval()` with the approval record. For urgent handoffs, includes WhatsApp notification.

c. Returns a `ToolResult` with:
   - `success: true`
   - `queued: true` (signals TAOR loop this is an async action)
   - `approvalId`: the created approval record ID
   - `data`: `{ status: 'handoff_requested', description: input.description, urgency: input.urgency || 'normal' }`

3. **Handoff resolution** â€” when user completes and confirms:
   - User resolves the approval via dashboard/WhatsApp/chat with a completion message
   - The existing `resolveApproval()` flow handles this
   - The `execution_result` field on the approval record carries the user's confirmation/result
   - If the handoff was part of an async task, the task resumes with the result

No new database table needed â€” reuses `approval_queue` table with `action_type = 'human_handoff'`.
