# Plan 42-02: ToolResolver — Context Injection & Tier-Aware Dispatch

<objective>
Implement the ToolResolver system that injects reliability data into model context and records execution outcomes from tool dispatch. The model carries the reasoning weight for tier selection (CONTEXT.md D-01); ToolResolver provides the data and records the results. This addresses CHAIN-01 (API/browser/workspace/human fallback) and CHAIN-02 (service-to-tier knowledge via reliability context).
</objective>

<tasks>

<task id="42-02-01">
<title>Create ToolResolver context enrichment module</title>

<action>
Create `personal-assistant/src/lib/agent/engine/tool-resolver.ts` with:

1. **`buildTierContextBlock()`** function:
```typescript
export async function buildTierContextBlock(
  supabase: SupabaseClient,
  orgId: string,
): Promise<string>
```

This function:
- Calls `getReliabilitySummary(supabase, orgId)` from reliability-tracker
- Calls `formatReliabilityContext(summaries)` to get the markdown table
- Wraps it with tier availability context:
```
## Available Execution Tiers

You have 4 execution tiers available for tasks. Choose the most appropriate tier based on the task, service, and reliability data below.

1. **Structured API** — Use existing tool integrations (Composio, channel tools, core tools). Cheapest and fastest. Preferred when available.
2. **Browser Automation** — Use `spawn_browser_agent` for websites without API integrations or when APIs are unreliable. Medium cost.
3. **Ephemeral Workspace** — Use `spawn_ephemeral_workspace` for compute-heavy tasks requiring custom scripts, data transforms, or package installation. Higher cost.
4. **Human Handoff** — Use `request_human_handoff` when the task requires physical world action, legal/financial sign-off, or when all automated tiers have failed. Pauses the task until the user confirms.

{reliability_table_or_empty}

When choosing a tier:
- Prefer structured API when the service has a known integration and good reliability
- Escalate to browser when API is unavailable or has low success rate for this service
- Use workspace for tasks that need custom code execution, not for service interactions
- Use human handoff as last resort or when the task inherently requires human judgment
```

Returns empty string if no tiers are available (should never happen in practice).

2. **`getTierForTool()`** helper:
```typescript
export function getTierForTool(toolName: string): 'api' | 'browser' | 'workspace' | 'human' | null
```
Maps tool names to their tier:
- `spawn_browser_agent` → `'browser'`
- `spawn_ephemeral_workspace` → `'workspace'`
- `request_human_handoff` → `'human'`
- All other tools → `'api'`
- Unknown/null → `null`

3. **`recordToolOutcome()`** wrapper:
```typescript
export async function recordToolOutcome(
  supabase: SupabaseClient,
  orgId: string,
  toolName: string,
  toolInput: Record<string, unknown>,
  success: boolean,
  error?: string,
  latencyMs?: number,
): Promise<void>
```
Combines `inferServiceName()` + `getTierForTool()` + `recordExecution()` into a single call. Fire-and-forget.
</action>
<acceptance_criteria>

- `getTierForTool('spawn_browser_agent')` returns `'browser'`
- `getTierForTool('send_email')` returns `'api'`
- `getTierForTool('request_human_handoff')` returns `'human'`
- `recordToolOutcome` calls `recordExecution` from reliability-tracker
</acceptance_criteria>

<title>Record execution outcomes in tool executor</title>
<read_first>
- personal-assistant/src/lib/agent/engine/tool-executor.ts (where tool results are processed)
- personal-assistant/src/lib/agent/engine/tool-resolver.ts (recordToolOutcome function)

<action>
Modify `tool-executor.ts` to record outcomes:

1. Import `recordToolOutcome` from `./tool-resolver`

2. In `executeToolBatchStreaming()`, after each tool execution result is processed (in the completion-order loop, after the `try` block processes the result), add:
```typescript
// Fire-and-forget reliability recording
recordToolOutcome(
  config.supabase,
  config.orgId,
  tool.name,
  tool.input as Record<string, unknown>,
  result.success,
  result.error,
  /* latencyMs: calculate from tool start time if available */
).catch(err => logger.debug('[engine] reliability recording failed', { err, tool: tool.name }))
```

3. Also add recording in the `executeToolBatch()` (non-streaming) function for consistency, in the same pattern.

4. For latency tracking, record the start time before each tool execution and calculate elapsed time. Add `const toolStartTimes = new Map<string, number>()` and record `Date.now()` before dispatch, then `Date.now() - start` after completion.
</action>
<acceptance_criteria>
- `tool-executor.ts` imports `recordToolOutcome` from `./tool-resolver`
- Both `executeToolBatch` and `executeToolBatchStreaming` call `recordToolOutcome` after tool completion
- Recording is fire-and-forget (`.catch()` pattern, no await blocking dispatch)
- Latency is tracked per-tool execution
</acceptance_criteria>
</task>
