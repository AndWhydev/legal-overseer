**use-file-upload.ts** -- Create a React hook managing the full upload lifecycle:

```typescript
interface UploadItem {
  id: string           // attachment ID from server
  filename: string
  mimeType: string
  size: number
  progress: number     // 0-100
  status: 'pending' | 'uploading' | 'ready' | 'error'
  previewUrl?: string  // Object.createObjectURL for local preview
  error?: string
}

interface UseFileUploadReturn {
  uploads: UploadItem[]
  addFiles: (files: FileList | File[]) => Promise<void>
  removeUpload: (id: string) => void
  clearUploads: () => void
  readyAttachmentIds: string[]  // IDs of uploads with status='ready'
  isUploading: boolean          // any upload in progress
}
```

Implementation details:
1. `addFiles` validates each file client-side using `validateFile()` from constants.ts. Rejects invalid files immediately with toast-style error (set item status='error' with error message). Enforces `MAX_FILES_PER_MESSAGE` cap across all current uploads.
2. For each valid file, call `POST /api/attachments/upload` with `{ filename, mimeType, size, threadId }` to get signed URL.
3. Upload file directly to signed URL via XHR (not fetch) to get `upload.onprogress` events. Update `progress` on each event.
4. On XHR success, call `PATCH /api/attachments/{id}` to confirm upload. Set status='ready'.
5. On XHR error, set status='error' with error message.
6. For image files, create a local preview URL via `URL.createObjectURL(file)` and store in `previewUrl`. Revoke on cleanup/remove.
7. `readyAttachmentIds` is a computed array filtering uploads where status='ready'.
8. `clearUploads` revokes all Object URLs and resets the array.

The hook must accept an optional `threadId` prop to pass to the upload API.

**content-blocks.ts** -- Create the module that builds Anthropic content blocks from attachment records:

```typescript
import type Anthropic from '@anthropic-ai/sdk'
import type { SupabaseClient } from '@supabase/supabase-js'
import { processAttachment } from '@/lib/rag/attachment-processor'
import { STORAGE_BUCKET, DOWNLOAD_URL_EXPIRY } from './constants'

interface AttachmentRecord {
  id: string
  filename: string
  mime_type: string
  size: number
  storage_path: string
  extracted_text: string | null
}

export async function buildAttachmentContentBlocks(
  supabase: SupabaseClient,
  attachments: AttachmentRecord[]
): Promise<Anthropic.ContentBlockParam[]>
```

Implementation:
1. For each attachment:
   - **Images** (mime_type starts with 'image/'): Generate signed download URL (1hr expiry), create `{ type: 'image', source: { type: 'url', url: signedUrl } }` block.
   - **PDFs** (mime_type === 'application/pdf'): Generate signed download URL, create `{ type: 'document', source: { type: 'url', url: signedUrl }, title: filename }` block. Use URL source (not base64) -- simpler and avoids downloading the file server-side.
   - **Text documents** (DOCX, CSV, TXT): If `extracted_text` exists on the record, use it. Otherwise, download from storage, call `processAttachment()` to extract text, update the record with extracted_text (fire-and-forget), then create `{ type: 'text', text: '[Attached file: ${filename}]\n\n${extractedText}' }` block.
2. Return the array of content blocks.

IMPORTANT: If signed URL generation fails for any attachment, skip it (log warning) rather than failing the entire message. Users should still be able to send text even if an attachment URL fails.
