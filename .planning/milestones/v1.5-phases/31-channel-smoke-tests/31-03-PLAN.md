**Monitoring API (GET /api/admin/monitoring):**

Requires authenticated admin user (check `profiles.role === 'admin'` or user ID matches org owner).

Returns JSON with 5 sections:

1. **cron_stats**: Query the activity_feed or build from cron response logs. For each of the 22 cron routes in vercel.json, return:
   - `route`: path (e.g., "/api/cron/channel-sync")
   - `last_run`: ISO timestamp of last successful execution (from activity_feed where action='cron_*' or from a new `cron_runs` table if one exists, otherwise derive from activity_feed)
   - `success_rate_24h`: percentage of successful runs in last 24h
   - `avg_duration_ms`: average execution time
   - `error_count_24h`: number of errors in last 24h

   Implementation: Query `activity_feed` where `action_type = 'system'` and `action LIKE 'cron_%'` grouped by action, last 24 hours. If activity_feed doesn't have cron execution logs, create a lightweight `cron_execution_log` table approach: add a log insert at the end of `withCronGuard` (both success and failure paths) into a `cron_execution_log` with columns: `id uuid PK, route text, success boolean, duration_ms int, error text null, created_at timestamptz default now()`. Use Supabase service client to insert. Retention: auto-delete rows > 7 days via a simple `DELETE FROM cron_execution_log WHERE created_at < now() - interval '7 days'` at the start of the monitoring query.

2. **agent_latency**: Query `agent_runs` table for last 24h:
   - `total_runs`: count
   - `p50_ms`, `p95_ms`, `p99_ms`: computed from `duration_ms` column (use SQL percentile_cont or fetch + sort in JS)
   - `error_rate`: percentage of runs with `status = 'error'`

3. **channel_health**: Query `channel_health` table (populated by health.ts storeHealthReports) or run `checkAllChannelHealth` inline:
   - Per channel: `channel`, `status`, `latencyMs`, `lastSync` (from `channel_connections.last_sync_at`), `error_count_24h` (from `channel_messages` error logs or activity_feed)

4. **error_summary**: Top 10 errors from `dead_letter_queue` in last 24h grouped by `agent_type`:
   - `agent_type`, `count`, `latest_error`, `latest_at`

5. **token_spend**: Query `agent_runs` for token counts grouped by org:
   - `org_id`, `org_name` (join profiles), `total_tokens_24h`, `total_cost_24h` (compute from token count * model rate)

All queries use service-role Supabase client. Response cached for 30 seconds (use `Cache-Control: max-age=30`).

**Smoke Test API (POST /api/admin/smoke-test):**

Requires authenticated admin user. Accepts optional `{ channel?: ChannelType }` body to test a specific channel.

- Imports `runAllSmokeTests` from `smoke-test-runner.ts` (31-01)
- Calls runner with service client and admin's org ID
- Returns `SmokeTestReport` JSON
- 60-second timeout (smoke tests make real network calls)
