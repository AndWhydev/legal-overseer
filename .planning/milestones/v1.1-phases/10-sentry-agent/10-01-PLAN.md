    Create sentry.ts implementing deterministic Sentry runtime with Supabase DI (client passed as first arg).

    Required exports:
    - runSentryTick(supabase, orgId, agentConfigId): loads active due watches and evaluates each watch.
    - evaluateWatch(supabase, watch, now): routes to watch-type-specific evaluator and returns { triggered, severity, summary, evidence }.
    - buildRemediationSuggestion(issueType, evidence): returns concrete operator advice (for example: endpoint 500 -> "check deployment health and rollback last release", repeated keyword error -> "inspect failing subsystem logs and restart worker", negative sentiment -> "send acknowledgment + propose recovery action").

    Detection rules (no new dependencies):
    - error_keyword watch: query recent channel_messages for configured keywords in subject/body after last_checked_at.
    - uptime watch: fetch endpoint URL from watch.conditions with timeout from conditions.timeout_ms (default 5000); treat non-2xx or timeout as issue.
    - negative_sentiment watch: query recent inbound client/lead messages and match configured negative patterns list (fallback default list such as refund/cancel/angry/urgent issue).

    On triggered watch:
    - insert sentry_alerts row with remediation_suggestion
    - set watch last_triggered_at and status='triggered'
    - set next_escalation_at to now + escalation_minutes

    On non-triggered watch:
    - keep status='active'

    For all processed watches:
    - update last_checked_at and next_check_at = now + interval_seconds

    Wire scheduler.ts so when an enabled agent_config has agent_type='sentry' and is due, scheduler calls runSentryTick instead of only writing placeholder output. Keep existing stateless tick behavior and never-throw style.

    Add vitest coverage for:
    - due-watch selection by interval/next_check_at
    - each watch type trigger/non-trigger behavior
    - remediation suggestion is non-empty and issue-specific
    - scheduler triggers runSentryTick for sentry configs
    - scheduler keeps existing behavior for non-sentry agents
