# Onboarding Redesign — Conversational World Model Reveal

**Date:** 2026-04-03
**Status:** Approved design, pending implementation plan
**Scope:** Complete redesign of the BitBit onboarding experience — UX, UI, and technical architecture

---

## Overview

The current onboarding is a 5-stage linear wizard (workspace → connections → sync → agents → value) that collects form data, runs a lightweight 30-second scan, and drops the user into an empty-feeling dashboard with a welcome chat message. The full intelligence pipeline (6-month crawl → Opus synthesis → world model ingestion) exists but only runs silently in the background — the user never sees its output.

The redesign replaces this with a **conversational onboarding** where BitBit talks to the user in real-time as it learns their world, culminating in an **interactive knowledge graph reveal** that shows every person, project, and financial relationship BitBit discovered. The dashboard chrome then animates in around the conversation, and the user lands in a fully populated product from day one.

## Design Principles

- **Zero forms** — No workspace setup, no industry picker, no agent selection screen. Everything is inferred from the data. If BitBit needs to confirm something, it asks conversationally.
- **BitBit decides** — Agents are activated automatically based on what the crawl reveals. The user is informed, not asked.
- **The product is the onboarding** — The user is already inside the dashboard from the start. Onboarding just populates it and reveals the chrome.
- **Show, don't describe** — The knowledge graph is proof that BitBit understands. Not a summary paragraph — a living, interactive map of the user's world.

## The Flow

### Phase 1 — The Chat Opens (0s)

Full-screen chat interface at `/onboard`. Monochrome dark theme using shadcn/ui components. No sidebar, no nav, no dashboard chrome visible — just the chat.

BitBit's first message appears immediately:

> "Hey — I'm BitBit. Give me access to your email and I'll figure out the rest."

A floating shadcn `Card` overlays the bottom-right corner with OAuth connection buttons (Gmail, Outlook, Calendar). The card does not block the chat — the conversation is alive, and the OAuth prompt is a companion element.

### Phase 2 — Connection + Crawl (~0s → ~30s)

User clicks an OAuth button, completes the redirect, and returns. The floating card dissolves with an animate-ui exit transition. BitBit starts narrating as the multi-channel crawl runs:

> "Connected. Reading through your last 6 months..."
> "Found someone called Steve West — 47 messages, looks like a client. Ring a bell?"

The user can respond naturally ("Yeah he's my biggest client"). Haiku acknowledges the correction and continues narrating. User corrections are tagged and folded into the world model during synthesis.

Narration messages are generated by Haiku consuming raw pipeline events (contacts found, threads identified, attachment text extracted) plus the conversation history so far. This is not templated — Haiku generates genuine BitBit-voice responses.

### Phase 3 — Opus Synthesis (~30s → ~2-3min)

The crawl finishes and Opus synthesis begins. BitBit's narration shifts from discovery to understanding:

> "Starting to see the bigger picture. You've got a few projects running — presaleservices.com.au keeps coming up with Steve..."
> "There's about $1,200 floating across a few invoices. One looks overdue."

The user can continue to interact — confirming, correcting, or adding context. Haiku processes these alongside the pipeline events.

During this phase, the world model ingestion runs in parallel: contacts are created/updated, the knowledge graph is populated (person + project + financial nodes, relationship edges), semantic memories are stored.

### Phase 4 — The Reveal (~2-3min)

Opus synthesis completes. BitBit delivers the reveal:

> "Here's your world as I see it. Tap anything that's wrong."

The **knowledge graph** renders inline in the chat stream — a force-directed graph (left) with a detail panel (right).

**Reveal animation sequence (total ~2s):**

1. **You** node fades in at center (0ms)
2. **Top 3 people** animate in from edges, relationship lines draw themselves (200ms stagger)
3. **Remaining people** cascade in, smaller nodes (400ms)
4. **Projects** slide in from below, edges connect to relevant people (600ms)
5. **Financials** pulse in with a brief yellow glow (800ms)
6. **Stats bar** counts up along the bottom: "23 people · 4 projects · $1,200 outstanding · 3 agents active"
7. Detail panel slides in showing the highest-frequency contact

BitBit also announces agent activation:

> "I've set up Finance to track that $1,200 in outstanding invoices, Comms is watching your inbox, and Sales is keeping an eye on your pipeline. You can adjust these anytime."

### Phase 5 — The Dashboard Transition

User clicks "Let's go" or responds affirmatively. The dashboard materializes around the existing chat:

1. **Graph scales down** — shrinks from full-screen to command center size (~400ms)
2. **Sidebar slides in from left** — nav items stagger in (50ms per item, ~300ms)
3. **Top nav fades down** — drops in with translate-y (200ms)
4. **Chat repositions** — shrinks to its dashboard tab position (the conversation persists)
5. **Tab bar appears** — chat tab active, other tabs (contacts, invoices, leads) visible with subtle pulse indicating they have data
6. **URL updates** — `router.replace('/dashboard')` via shallow routing, no page reload

Total transition: ~1.5 seconds.

**What the user lands on:** A fully populated dashboard. Contacts tab has people with relationships and message counts. Invoices tab has outstanding financials as drafts. Chat tab has the onboarding conversation. Command center has the knowledge graph. Agents show Finance, Comms, Sales active with first observations. No empty states.

## Technical Architecture

### Component 1: Onboarding Conversation Service (Fly.io Worker)

A streaming SSE endpoint on the existing Fly.io worker (`bitbit-workers.fly.dev`), not Vercel — the pipeline needs 2-5 minutes, exceeding Vercel's 300s limit.

**Endpoints:**

- `POST /api/onboarding/conversation` — Initiates the pipeline and opens an SSE stream. Authenticated via bearer token + user session forwarded from Vercel.
- `POST /api/onboarding/conversation/reply` — Accepts user messages during onboarding. Haiku processes them and the next SSE narration event reflects the input.

**SSE event types:**

| Event | Payload | Frontend Action |
|-------|---------|-----------------|
| `narration` | `{ message: string }` | Render as BitBit chat bubble |
| `discovery` | `{ type: 'contact'\|'project'\|'financial', data: {...} }` | Show inline mention in chat |
| `progress` | `{ phase: string, percent: number }` | Update subtle progress indicator |
| `reveal` | `{ worldModel: WorldModel, stats: {...} }` | Render knowledge graph |
| `agents` | `{ activated: string[], reasons: Record<string, string> }` | Show agent activation message |
| `complete` | `{ threadId: string }` | Enable "Let's go", save conversation |
| `error` | `{ message: string, recoverable: boolean }` | Show error with retry option |

**Pipeline flow inside the worker:**

1. Run `crawlAllChannels()` (existing `intelligence-crawl.ts`) — emits `progress` + raw crawl events
2. Feed each raw event + conversation history to Haiku → emit `narration` + `discovery` events
3. Run `synthesizeWorldModel()` (existing `opus-synthesis.ts`) — emits `progress`
4. Run `ingestWorldModel()` (existing `world-model-ingester.ts`) — populates DB
5. Auto-activate agents based on world model content → emit `agents`
6. Emit `reveal` with full world model
7. Save conversation to `conversation_threads` + `conversation_messages` → emit `complete`

**Model usage:**

- **Haiku** — Narration generation. Receives: raw pipeline event + conversation history + user corrections. Generates: next BitBit message in SOUL.md voice. Cost: ~$0.01-0.03 per onboarding.
- **Opus** — World model synthesis. Single large call with full corpus. Cost: ~$2-5 per onboarding. Already implemented in `opus-synthesis.ts`.

### Component 2: Onboarding Chat UI (Next.js)

A new page component at `src/app/(auth)/onboard/page.tsx` — complete rewrite of the current 958-line wizard.

**UI framework:** shadcn/ui components + animate-ui for transitions. No inline glassmorphism styles.

**Key components:**

- **`OnboardingChat`** — Full-screen chat container. Connects to SSE stream. Renders narration events as chat bubbles. Handles user input via the reply endpoint. Uses shadcn `ScrollArea`, `Input`, `Button`.
- **`ConnectionCard`** — Floating shadcn `Card` with OAuth buttons. Positioned bottom-right via absolute positioning. Dissolves on successful connection. Uses animate-ui `FadeIn`/`FadeOut`.
- **`WorldGraph`** — The knowledge graph component (see below). Renders inline in the chat stream when the `reveal` event arrives.
- **`DashboardShell`** — The sidebar, nav, and tab bar components. Always mounted but hidden (`opacity: 0`, `pointer-events: none`) during onboarding. Animate in on completion.

**State management:** React state + `useEffect` for SSE connection lifecycle. No external state library needed.

### Component 3: Knowledge Graph

**Rendering:** `d3-force` for physics simulation, rendered to `<canvas>` for performance with 50+ nodes.

**Node types:**

| Type | Shape | Border Color | Size Logic |
|------|-------|-------------|------------|
| You (center) | Circle | White (rgba 255,255,255,0.2) | Fixed, largest |
| Person | Circle | Green (semantic, 12% opacity) | Scaled by message count |
| Project | Rounded rectangle | Blue (semantic, 12% opacity) | Fixed medium |
| Financial | Rounded rectangle | Yellow (semantic, 12% opacity) | Fixed small |

**Edges:**

- Solid lines: direct relationships (contacted, works-on)
- Dashed lines: inferred relationships (shared project membership)
- Opacity: scaled by recency of last interaction
- Width: uniform thin (1-1.5px)

**Interactions:**

- Click node → select (highlights node + connected edges, opens detail panel)
- Drag node → reposition
- Scroll → zoom
- Double-click → pin/unpin node position

**Detail panel:** shadcn `Sheet` or side panel anchored right. Content varies by node type:

- **Person:** Name, company, role, relationship, message count, frequency, last interaction, outstanding items, linked projects, linked financials
- **Project:** Name, status, people involved, URLs/domains, deadlines, blockers
- **Financial:** Type, entity, amount, due date, status, reference

Each field is **editable inline**. Corrections fire a `PATCH` to update contacts/memories in Supabase.

**Data mapping from WorldModel:**

```
nodes:
  WorldModel.user           → center "You" node
  WorldModel.people[]       → person nodes (circle, green)
  WorldModel.projects[]     → project nodes (rounded rect, blue)
  WorldModel.financials[]   → financial nodes (rounded rect, yellow)

edges:
  you → person              (from person.communicationFrequency != 'rare')
  person → project          (from project.people[] matching person.name)
  person → financial        (from financial.entity matching person.name/company)
  person → person           (from shared project.people[] membership)
```

### Component 4: Conversation Persistence

When onboarding completes (user clicks "Let's go"):

1. All narration messages + user replies written to `conversation_threads` + `conversation_messages` as the first real thread
2. User corrections stored as high-confidence `semantic_memories`
3. World model already ingested by pipeline (contacts, kg_nodes, kg_edges, semantic_memories, entity_profiles)
4. `profile.preferences.onboarding_completed = true` set via existing PATCH endpoint
5. Activated agents written to `role_configs` with appropriate autonomy levels

The onboarding conversation persists as the first thread in the chat tab — it's not ephemeral.

### Post-Onboarding Graph Access

The knowledge graph component remains accessible from the dashboard — embedded in the command center tab as a "Your World" section. It updates as new data flows in through regular channel sync and intelligence crons.

## Error Handling

| Failure | User Experience | Technical |
|---------|----------------|-----------|
| OAuth fails | BitBit: "Hmm, couldn't connect. Want to try again?" + OAuth card reappears | Retry with fresh OAuth URL |
| Crawl returns 0 messages | BitBit: "Your account looks pretty quiet. Try connecting a different one?" + OAuth card | Pipeline aborts gracefully, no synthesis |
| Crawl partially fails (1 channel errors) | BitBit continues with available data, mentions the gap | `Promise.allSettled` already handles this |
| Opus synthesis fails | BitBit: "Hit a snag building your world model. Let me try again." Auto-retry once | Retry with truncated corpus if token limit |
| Haiku narration fails | Fall back to templated messages (existing discovery line format) | Graceful degradation, pipeline continues |
| SSE connection drops | Frontend auto-reconnects with `Last-Event-ID` | SSE reconnection with event replay |
| User closes browser mid-onboarding | On return, resume from last pipeline phase if still running, or restart | Pipeline state persisted to Supabase |
| Worker timeout (>5min) | BitBit: "This is taking longer than expected. I'll keep working — check back in a minute." | Pipeline continues, frontend polls for completion |

## Success Criteria

1. New user goes from signup to fully populated dashboard in under 5 minutes
2. The knowledge graph shows real people, projects, and financials from the user's email — not placeholder data
3. User can correct any piece of data during or after the reveal
4. No empty states on first dashboard load — contacts, invoices, and chat all have data
5. The dashboard transition feels cinematic, not jarring — the chrome materializes around the conversation
6. Agent activation happens automatically with sensible defaults based on the world model
7. The onboarding conversation persists as the first chat thread
8. Zero forms filled out by the user (name, business, industry all inferred)
